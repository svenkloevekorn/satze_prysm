🛒
Shopify braucht SKUs

Jede verkaufbare Einheit muss eine eindeutige SKU haben. Ohne das geht keine Bestellung.

📦
Bestand pro Variante

„M Schwarz" ist ausverkauft, „M Weiß" hat 50 Stück → muss separat gepflegt werden.

💶
Variante-Preise

XL kostet manchmal mehr (mehr Stoff). Sondereditionen haben Aufpreis. Pro Variante setzbar.

📉
Verkaufs-Statistiken

„Welche Größe verkauft sich am besten?" → ohne Varianten unbeantwortbar.

Mit Prysm Variantenmanagement

Pro Produkt eine Varianten-Matrix mit auto-generierten SKUs

Größen × Farben (× weitere Achsen wenn nötig). Klick → 10 Varianten in einer Sekunde generiert.

SKU-Schema (auto-generiert)
STZ-JRY-PSJ-M-BLK
STZ = Marke (Staeze)
JRY = Kategorie (Jersey)
PSJ = Produkt-Slug
M = Größe
BLK = Farbe

Schema in den Admin-Settings konfigurierbar. SKU bleibt eindeutig & menschenlesbar.

Was du genau bekommst

Module im Detail

📋
ProductVariant (polymorph)

Tabelle product_variants, polymorph an FinalProduct + SupplierProduct + CompetitorProduct. SKU, Größe, Farbe, Override-Preis.

⚡
SKU-Generator

Konfigurierbares Schema (Marke-Kategorie-Slug-Größe-Farbe). Auto-Generiert, einmal-eindeutig, in Settings anpassbar.

✨
Matrix-Generator

„Größen XS-XXL × Farben Schwarz/Weiß" → ein Klick → 14 Varianten. Statt 14× von Hand anlegen.

📦
Lager-Bestand

Pro Variante: aktueller Bestand, ggf. reservierte Stück. Auto-Status: Verfügbar / Knapp (< 10) / Ausverkauft.

💸
Preis-Override

Standard-VK kommt vom Produkt. Pro Variante optional ein Override (z.B. „XL kostet 5 € mehr").

🔌
Shopify-Bridge-Ready

SKU-Format kompatibel zu Shopify. Spätere Bridge kann 1:1 mappen, Bestand syncen, Preise pushen.

Warum gerade jetzt

Das öffnet die Tür für alles Verkaufs-Verwandte

1
Voraussetzung für Shopify

Du hast dich für Shopify als ersten Shop entschieden. Ohne SKU-Varianten kann der Shop nicht starten – das ist der Brückenkopf.

2
Bestandsführung von Tag 1

Auch ohne Shop kannst du sofort sagen: „M Schwarz haben wir noch 30 Stück, neue Bestellung beim Lieferanten lohnt".

3
Daten-Modell wird sauberer

Aktuell sind Größen/Farben unstrukturierte JSON-Arrays. Mit Varianten werden sie echte Datensätze – filterbar, abfragbar.

4
Einsicht in Sortiments-Performance

Sobald Shop läuft: „Welche Größe verkauft sich am besten?", „Welche Farbe ist Ladenhüter?" – klare Daten statt Bauchgefühl.

Realistisch

Aufwand & Phasen

MVP – nutzbar nach
Phase 1 · 1–2 Sessions
Empfohlener Start
  • ✓ Migration product_variants (polymorph)
  • ✓ SKU-Generator (Schema in Settings konfigurierbar)
  • ✓ Matrix-Generator („Größen × Farben → erstellen")
  • ✓ RelationManager „Varianten" im FinalProduct + SupplierProduct
  • ✓ Lager-Status (Bestand, automatischer Status verfügbar/knapp/aus)
  • ✓ Preis-Override pro Variante
Phase 2 · 1 Session
Bestand-Bewegungen + Reports
  • → Bestand-Bewegungen (Eingang vom Lieferanten, Korrektur, Verkauf manuell)
  • → Bestand-History (wie viele Stück wann da)
  • → Dashboard-Widget „Niedrigster Bestand" (Re-Order-Liste)
Phase 3 · separates Projekt
Shopify-Bridge
  • → Shopify-API anbinden, Produkte + Varianten anlegen
  • → Bestand sync: Prysm → Shopify
  • → Bestellungen sync: Shopify → Prysm (Lager-Abbau)

Phase 3 nur, wenn der Shopify-Account live ist und Sortiment groß genug.

💡 Mein Vorschlag

Phase 1 sehr bald angehen. Es ist überschaubar und schließt das wichtigste Loch Richtung Verkauf. Phase 2 + 3, wenn der Shop wirklich läuft.
